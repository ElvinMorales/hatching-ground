# Public/private boundary

## Classifications

- **Private-only:** The value depends on sensitive, identifying, licensed, confidential, or employer-controlled inputs. Keep the idea card and all working artifacts local and ignored.
- **Public-safe:** The concept, templates, and examples contain only synthetic or explicitly public material and reveal no private context.
- **Potentially publishable later:** The pattern could be public, but current artifacts need redaction, replacement, permission review, or a clean-room synthetic rewrite.

## Public-release checklist

- [ ] All examples are synthetic and labeled.
- [ ] No real personal, family, health, financial, or employer data appears.
- [ ] No private logs, memory, state, communications, or source exports appear.
- [ ] No screenshots of private systems appear.
- [ ] No API keys, tokens, passwords, credentials, or secret-bearing configuration appears.
- [ ] No machine-specific paths, usernames, or identifying metadata appears.
- [ ] Rights and permissions for every included artifact are clear.
- [ ] Git history and generated files have been inspected, not only the latest diff.
- [ ] Another human has reviewed the release boundary.

Real personal data, family data, health data, financial data, employer data, private logs, real memory, real state, screenshots of private systems, API keys, and tokens are explicitly banned from public examples.
