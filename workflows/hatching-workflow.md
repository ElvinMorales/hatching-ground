# Hatching workflow

1. Confirm the selected idea card reflects the intended problem.
2. Review score evidence and resolve any blocking uncertainty.
3. Shrink the MVP to one user, input-to-output loop, and first-run test.
4. Classify data and public/private boundaries.
5. Run the hatch gate below.
6. If every gate passes, copy and complete the architecture brief.
7. Review deferred features and open questions.
8. Prepare a Codex handoff for a separate, explicitly authorized implementation pass.

## Hatch gate

- [ ] Problem is clear.
- [ ] User is clear.
- [ ] Smallest useful version is clear.
- [ ] Privacy boundary is clear.
- [ ] Likely artifacts are known.
- [ ] First-run scenario is testable.
- [ ] No high-risk automation is required.

If a gate fails, clarify, shrink, split, or park the idea. Do not disguise missing decisions as architecture assumptions.

The architecture brief is pre-architecture. Producing a full architecture, implementation plan, or runtime is a separate step after human review.
